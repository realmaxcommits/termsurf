# Issue 755: Surfari — WebKit engine for TermSurf

## Goal

Build Surfari, the WebKit-based browser engine for TermSurf. Surfari proves that
the TermSurf protocol is truly engine-agnostic by running a second browser
engine alongside Roamium (Chromium) in the same terminal window.

## Background

### Why WebKit first

TermSurf claims to be a protocol that works with any engine, but it currently
only runs Chromium. Shipping a second engine proves the architecture. WebKit is
the best candidate because:

- **Potentially zero fork modifications.** WKWebView is a public API. We can
  link system `WebKit.framework` on macOS without building WebKit from source.
- **Smallest C library.** Estimated ~500 lines — the smallest of all four
  engines.
- **Native compositing.** WKWebView is an NSView. We position it as an overlay
  in the terminal window and let Window Server composite it. No CALayerHost, no
  GPU buffer extraction — simpler than Chromium.
- **Public APIs for everything.** Multi-profile (`WKWebsiteDataStore`), DevTools
  (`_WKInspector`), navigation callbacks (KVO), input (NSView first responder).
- **Prior art.** TermSurf 1.x (ts1) already implemented a full WKWebView
  integration with Ghostty, including navigation, DevTools, profiles, downloads,
  and console capture.

### Architecture

Surfari follows the same pattern as Roamium:

```
libtermsurf_webkit (Objective-C, C API)
        ↓ links
    Surfari (Rust binary, ~400 lines)
        ↓ connects via Unix socket
    Wezboard (GUI)
```

- `libtermsurf_webkit` — Objective-C library wrapping WKWebView, exporting C
  functions with `ts_*` signatures matching the same API as
  `libtermsurf_chromium`.
- Surfari — Rust binary handling Unix socket IPC, protobuf parsing, and process
  lifecycle. Almost entirely reusable from Roamium.

### Compositing: NSView overlay vs CALayerHost

Chromium uses CALayerHost for zero-copy GPU compositing. WebKit does not need
this. WKWebView is an NSView — it composites through macOS's standard view
hierarchy. The approach:

1. Surfari process creates a hidden `NSWindow` containing a `WKWebView`
2. The GUI (Wezboard) receives the rendered content via... what exactly?

This is the key open question. Unlike Chromium, where the CALayerHost ID crosses
the process boundary over a socket, WKWebView's rendering is tied to its NSView.
Two possible approaches:

**Approach A: Reparent the WKWebView into Wezboard's window.** The Surfari
process creates the WKWebView, then the GUI reparents it into the terminal
window as a subview at the correct pixel coordinates. This requires
cross-process NSView reparenting, which macOS does not support directly. The
view and its window must be in the same process.

**Approach B: Create the WKWebView in the GUI process.** Instead of Surfari
owning the WKWebView, the GUI (Wezboard) creates it directly. Surfari becomes a
thin control process that sends navigation commands over the socket, while the
actual WKWebView lives in Wezboard. This changes the architecture — the engine
process no longer owns the rendering surface.

**Approach C: Extract a CAContext from WKWebView's layer tree.** WKWebView has
an internal layer tree. We could create a `CAContext` from its root layer and
export the `contextId` to Wezboard, just like Chromium. This would require ~20
lines of modification to WebKit's `LayerHostingContext.mm` — but it would keep
the architecture uniform with Roamium.

**Approach D: Use WKWebView's off-screen rendering.** Render WKWebView into a
bitmap and send the pixels. This is the CEF approach that we already rejected
for performance reasons.

### Which approach?

Approach C is the most architecturally consistent — it keeps the same
CALayerHost pattern that Roamium uses, the same process model, and the same
Wezboard code path. The modification to WebKit is small. The downside is that it
requires building WebKit from source instead of linking the system framework.

Approach B is the simplest — no WebKit modifications, no build-from-source. But
it breaks the architectural principle that the engine process owns its rendering
surface.

This needs research and experimentation to determine which approach is viable.

### Multi-profile

WKWebView supports multiple profiles in one process via `WKWebsiteDataStore`.
However, TermSurf uses one-process-per-profile for architectural uniformity
across all engines. Surfari will follow the same model.

```objc
WKWebsiteDataStore *store = [WKWebsiteDataStore
    dataStoreForIdentifier:[[NSUUID alloc] init]];
config.websiteDataStore = store;
```

### Input handling

WKWebView is an NSView. macOS routes keyboard and mouse events to the first
responder automatically. If the WKWebView is in the same process as the GUI
(Approach B), input routing is free. If it's in a separate process (Approach C),
we forward events over the socket just like Roamium.

### DevTools

WebKit uses Web Inspector, not Chrome DevTools:

```objc
[webView._inspector show];       // open inspector
[webView._inspector attachRight]; // dock right
```

### Prior work from ts1

Issue 108 documented a full WKWebView integration in TermSurf 1.x including:
navigation delegates, download handling, console capture via JS injection,
profile isolation, DevTools, HTTP auth, dialogs, file uploads, and crash
recovery. Much of this code can inform Surfari's implementation.

### What needs to happen

1. Determine the compositing approach (Approach B vs C — research needed)
2. Create `libtermsurf_webkit` (Objective-C library with `ts_*` C API)
3. Create the Surfari Rust binary (fork from Roamium, adapt)
4. Update Wezboard to launch Surfari processes for WebKit profiles
5. Update the `web` TUI to support `--browser webkit` (or similar)
6. Test with all existing protocol messages
